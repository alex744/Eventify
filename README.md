# Eventify

- **API для управления событиями (CRUD операции) и бронированиями.**

## Требования

- [.NET 10 SDK](https://dotnet.microsoft.com/ru-ru/download/dotnet/10.0) 

## Запуск проекта

1. Клонируйте репозиторий:
   ```bash
   git clone https://github.com/alex744/Eventify.git
   cd Eventify\src
   ```

2. Восстановите зависимости:
   ```bash
   dotnet restore
   ```

3. Соберите проект:
   ```bash
   dotnet build
   ```

4. Запустите веб‑приложение:
   ```bash
   dotnet run --project Ya.Events.WebApi
   ```

5. При необходимости запустите тесты:
   ```bash
   dotnet test
   ```

6. Приложение будет доступно по адресам:
   - `http://localhost:5000`
   - `https://localhost:5001`

7. Документация Swagger:
   - `http://localhost:5000/swagger`

## Краткая документация API

Все эндпоинты по базовому пути `/events` (кроме получения бронирования). 

### Модель Event

```json
{
  "id": "guid",
  "title": "string",
  "description": "string | null",
  "startAt": "2025-06-01",
  "endAt": "2025-06-02",
  "totalSeats": 100,
  "availableSeats": 100
}
```
**Поля:**
- `id` - уникальный идентификатор события.
- `title` - название события.
- `description` - описание события.
- `startAt` - дата начала события.
- `endAt` - дата окончания события.
- `totalSeats` – общее количество мест на событии (обязательное, положительное целое).
- `availableSeats` – количество ещё доступных для бронирования мест. При создании события равно totalSeats. Уменьшается на 1 при каждой успешно созданной брони, увеличивается на 1 при возврате места (отклонении брони).

**Валидация:** 
- `title` обязательно.
- `endAt` позже `startAt`.
- `totalSeats` ≥ 1.

### Эндпоинты

| Метод | Путь                | Описание                                | Основные ответы                      |
|-------|---------------------|-----------------------------------------|--------------------------------------|
| GET   | `/events`           | Получить все события (с фильтрацией)    | 200: Список событий                  |
| GET   | `/events/{id}`      | Получить событие по `id` (GUID)         | 200: Событие, 404: Not Found         |
| POST  | `/events`           | Создать новое событие                   | 201: Created + заголовок Location    |
| PUT   | `/events/{id}`      | Обновить событие (полный объект)        | 204: No Content, 404: Not Found	   |
| DELETE| `/events/{id}`      | Удалить событие по `id` (GUID)          | 204: No Content, 404: Not Found      |
| POST	| `/events/{id}/book` | Забронировать событие				    | 202: Accepted + тело бронирования	   |
| GET	| `/bookings/{id}`	  | Получить информацию о бронировании	    | 200: Бронирование, 404: Not Found	   |

**Важно:** эндпоинт `POST /events/{id}/book` теперь может возвращать `409 Conflict` при отсутствии свободных мест (см. раздел «Примитивы синхронизации»).

### Фильтрация и пагинация для GET /events

Для `GET /events` поддерживаются параметры запроса вида:
`/events?title=Конференция&from=2025-06-01&to=2025-06-30&pageSize=10&page=2`

#### Параметры фильтрации

| Параметр   | Тип        | Описание																																   |
|------------|------------|--------------------------------------------------------------------------------------------------------------------------------------------|
| `title`    | `string`   | Частичный поиск по названию события (без учёта регистра). Если не задан — фильтр не применяется.										   |
| `from`     | `datetime` | Возвращает только события, у которых `startAt` ≥ указанной даты‑времени. Если не задан — ограничение по началу интервала не накладывается. |
| `to`       | `datetime` | Возвращает только события, у которых `endAt` ≤ указанной даты‑времени. Если не задан — ограничение по концу интервала не накладывается.    |

#### Параметры пагинации

| Параметр   | Тип    | Описание																									 |
|------------|--------|--------------------------------------------------------------------------------------------------------------|
| `pageSize` | `int`  | Количество событий на одной странице (по умолчанию, 10). Если не задан — используется значение по умолчанию. |
| `page`     | `int`  | Номер страницы (начиная с 1). Если не задан — возвращается первая страница.									 |

### Примеры запросов GET /events

- Все события с названием, содержащим «Конференция»:
  `/events?title=Конференция`

- События в июне 2025 года:
  `/events?from=2025-06-01&to=2025-06-30`

- Пагинация: 10 событий, вторая страница:
  `/events?pageSize=10&page=2`

- Смешанный запрос:
  `/events?title=Доклад&from=2025-06-01&page=1`

При некорректных значениях (неверный формат даты, `from` позже `to`, отрицательные/нулевые числа в `pageSize`/`page`) API возвращает `400 Bad Request` с описанием ошибки.

### Пример POST‑запроса

```json
{
  "title": "Конференция",
  "description": "Описание (необязательно)",
  "startAt": "2025-06-01",
  "endAt": "2025-06-01",
  "totalSeats": 100
}
```

### Бронирование событий

### Модель Booking
```json
{
  "id": "guid",
  "eventId": "guid",
  "status": "Pending"
  "createdAt": "2025-04-08T10:00:00Z",
  "processedAt": "2025-04-08T10:00:00Z | null"
}
```

**Поля:**
- `id` – уникальный идентификатор бронирования.
- `eventId` – идентификатор события, на которое бронируется место.
- `status` – текущий статус бронирования (см. ниже).
- `createdAt` – дата и время создания брони.
- `processedAt` – дата и время обработки брони (опционально).

### Статусы бронирования

| Статус	  | Описание						  |
|-------------|-----------------------------------|
| `Pending`	  | Бронь создана, ожидает обработки. |
| `Confirmed` | Бронь подтверждена.				  |
| `Rejected`  | Бронь отклонена.				  |

### Фоновая обработка бронирований

- После создания бронирования (статус `Pending`) запускается фоновый сервис, который через заданный интервал (3 секунды) подтверждает бронирование, меняя его статус на `Confirmed`. Если событие было удалено до подтверждения, бронь автоматически отклоняется.
- Все изменения статусов логируются и доступны через `GET /bookings/{id}`.

### Примитивы синхронизации

Для предотвращения овербукинга (превышения доступных мест при параллельных запросах) в `BookingService` используется `SemaphoreSlim` (асинхронный семафор с начальным и максимальным счётчиком, равным 1).
- Критическая секция, защищённая семафором, включает атомарную операцию «проверка свободных мест + резервирование» (`TryReserveSeats()`). Только один поток может одновременно выполнять эту операцию для конкретного события.
- Благодаря этому при 5 свободных местах и 20 одновременных запросах ровно 5 получат успешное бронирование, а 15 – ответ `409 Conflict` с исключением `NoAvailableSeatsException`.
- Семафор не блокирует потоки, а использует асинхронное ожидание (`WaitAsync`), что важно для масштабируемости веб-приложения.

### Пример сценария бронирования

1.Пользователь находит событие через `GET /events`.
2.Отправляет `POST /events/{eventId}/book`.
3.API возвращает `202 Accepted` и бронирование со статусом `Pending`.
4.Фоновый сервис через 3 секунды подтверждает бронирование – статус становится `Confirmed`.
5.Пользователь может в любой момент проверить статус через `GET /bookings/{id}`.

### Пример сценария с овербукингом

1.Создано событие с `totalSeats = 5`.
2.Десять пользователей одновременно отправляют `POST /events/{eventId}/book`.
3.API возвращает первым пяти запросам статус `202 Accepted` и создаёт брони со статусом `Pending`.
4.Остальные пять запросов получают `409 Conflict` с сообщением `"No available seats for this event"`.
5.Фоновый сервис параллельно подтверждает пять созданных броней, статус каждой меняется на `Confirmed`.

### Формат ответов об ошибках

При ошибках валидации или бизнес‑логики возвращается `400 Bad Request` с JSON следующего вида:

```json
{
  "status": 404,
  "detail": "Событие с идентификатором '9e1514cf-90f7-46d6-9e30-8df604d59f58' не найдено."
}
```
